Deployment Rollback 机制验证：从失效场景到回滚边界

April 14, 2026 · 1 min
Table of Contents
  • 实验目标
  • 为什么记这个
  • 环境信息（可复现）
  • 新建实验环境
  • 实验过程

实验目标#

验证 Deployment 的 rollout undo 在以下场景的有效性：

  1. 正常回滚：镜像版本回退
  2. 中断恢复：更新过程中强制中断（模拟网络抖动）
  3. 历史边界：revision 被 GC 后的回滚行为

为什么记这个#

之前使用或者看官方文档都是用现成的yaml文件 kubectl apply，现在发现rollout配合YAML里的几个参数，能在发版炸了的瞬间实现秒级回滚，实用。

环境信息（可复现）#

  • K8s 版本: v1.35.3 (kubectl client v1.35.3)
  • 集群规模: 一主两从测试环境
  • 关键配置:
revisionHistoryLimit: 10      # 设置最多留存历史修改版本方便测试GC，回滚时有选择, 如果为0则不记录历史版本
strategy:
  rollingUpdate:
    maxSurge: 25%            # 更新时允许超出的比例
    maxUnavailable: 25%      # 最大不可用比例
terminationGracePeriodSeconds: 30  # 优雅停机， pod删除时最大宽限时间

新建实验环境#

–record 的作用是把"你执行了什么命令"记录到 Deployment 的元数据里，方便事后追溯 初始状态

kubectl scale deployment  nginx-deployment --replicas=3 
kubectl rollout history deployment/nginx-deployment # 输出: revision 1 (原始部署)

REVISION  CHANGE-CAUSE
1         kubectl scale deployment nginx-deployment --replicas=3 --record=true

实验过程#

将镜像指向v1.0版本并验证revision修改记录是否生成

# 初始：my-nginx (rev 1)
kubectl set image deployment/nginx-deployment hello-world-nginx=quay.io/redhattraining/hello-world-nginx:v1.0 --record

验证revision是否计入元数据

kubectl rollout history deployment/nginx-deployment

查看详细信息

 kubectl rollout history deployment/nginx-deployment --revision=1

生成新的revision

kubectl set image deployment/nginx hello-world-nginx=quay.io/redhattraining/hello-world-nginx:latest --record
